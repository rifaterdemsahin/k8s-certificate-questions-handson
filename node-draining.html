Node Draining, From First Principles

Why a cluster needs to drain a node, how the mechanism actually works, and the real terminal output from running it on a 2-node minikube cluster.

Why we need it — first principles

Start from one unavoidable fact: physical machines need to stop. Kernel patches, OS upgrades, a kubelet bump, a failing disk, or simply scaling the cluster down — every node will, sooner or later, have to go offline.

Now reason forward from what a node is: a box running pods. If you just power it off, every pod on it dies instantly and simultaneously. No SIGTERM, no time to finish in-flight requests, no time for a replacement to come up first. That is an involuntary disruption — an outage.

Kubernetes' core contract is declarative reconciliation: you declare "I want 4 replicas," and controllers continuously make reality match. That contract can heal a node that dies unexpectedly — but only after the fact, with a gap of downtime. The insight behind draining is to turn an unplanned, all-at-once node death into a planned, orchestrated, one-pod-at-a-time migration — a voluntary disruption the cluster can pace and protect.

To do that safely, you decompose the problem into four independent primitives:

1. Stop new work

Mark the node unschedulable so the scheduler stops placing new pods on a node that's about to leave. (cordon)

2. Move existing work

Evict the pods already running there. Controllers recreate them elsewhere — desired state is preserved. (drain)

3. Guarantee availability

A PodDisruptionBudget caps how many replicas may be down at once, so eviction never drops you below a safe floor.

4. Shut down gracefully

Each pod gets SIGTERM + a grace period to finish in-flight work before SIGKILL. No dropped requests.

The whole design in one line: separate "stop sending new work" from "move existing work," then gate the move with an availability budget and graceful termination. That's all draining is.

How it works — the mechanism

kubectl drain is really cordon + graceful eviction of every evictable pod. Here is the exact sequence the cluster runs:

  1. Cordon — node becomes Ready,SchedulingDisabled

    The scheduler is told: place no new pods here. Existing pods keep running, untouched, for now.

  2. Eviction API is called for each pod

    Unlike a raw delete, the eviction API checks PodDisruptionBudgets first. If evicting a pod would breach the budget, the request is rejected and retried — eviction proceeds only as fast as replacements become Ready.

  3. Graceful termination per pod

    Each evicted pod receives SIGTERM, runs any preStop hook, then has terminationGracePeriodSeconds to exit cleanly before SIGKILL.

  4. Controllers recreate the pods elsewhere

    The Deployment/ReplicaSet notices replicas are missing and the scheduler places fresh pods on the remaining schedulable node(s).

  5. Uncordon when maintenance is done

    kubectl uncordon flips the node back to schedulable so it can host pods again.

What governs the eviction

MechanismRole during a drain
Graceful terminationSIGTERM → preStop → terminationGracePeriodSeconds → SIGKILL
PodDisruptionBudgetCaps how many replicas may be down at once → keeps the app available
Controllers (Deploy/RS/STS)Recreate evicted pods on schedulable nodes
DaemonSet podsSkipped unless --ignore-daemonsets (they belong to every node)
Bare / unmanaged podsBlock the drain unless --force (nothing would recreate them)

The objects we apply

A 4-replica Deployment plus a PDB with minAvailable: 3 — so at most one pod may be voluntarily disrupted at a time:

# pdb.yaml — the availability guarantee that makes drain safe
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: draining-demo-pdb
spec:
  minAvailable: 3        # of 4 replicas → only 1 may be down at once
  selector:
    matchLabels:
      app: draining-demo
# the drain command (from run.sh)
kubectl drain "$NODE" \
  --ignore-daemonsets \      # skip kube-proxy / CNI pods tied to the node
  --delete-emptydir-data \   # allow evicting pods using emptyDir
  --force \                  # evict controllerless pods (e.g. storage-provisioner)
  --timeout=120s

Read the proof: the red lines are the PodDisruptionBudget doing its job — refusing to evict a pod while it would drop the app below minAvailable: 3, retrying every 5s until a replacement is Ready on m02. The node ends as SchedulingDisabled, all four pods land on the healthy node, and uncordon restores it. Zero availability dip.
Two real bugs this run surfaced (and the fixes):
  • VirtualBox auto-selected → unsupported on Apple Silicon. run.sh now pins --driver docker (override with DRIVER=…).
  • Drain aborted on the controllerless storage-provisioner pod. Added --force to the drain command.

Run it yourself

# 2-node cluster so evicted pods have somewhere to go
cd 5_Symbols/01_node_draining
./run.sh                      # cordon → drain → reschedule → uncordon

# cleanup
kubectl delete -f pdb.yaml -f deployment.yaml
minikube delete -p k8s-cert-qa
💡 With a single-node cluster the drain has nowhere to move pods, so the PDB blocks eviction forever — which is exactly why the lab uses --nodes 2.
